# Release process

How to cut a release of cluster-api-provider-ovhcloud.

## Versioning

The project follows [Semantic Versioning](https://semver.org/) starting
at v0.1.0. Until v1.0.0:

- `v0.x.0` → minor: new features, possibly minor breaking changes
- `v0.x.y` → patch: bug fixes, no breaking changes

After v1.0.0, breaking changes only on major version bumps.

## Pre-release checklist

Before tagging, verify:

- [ ] All open milestone issues are resolved or moved
- [ ] Local checks pass: `make verify test`
- [ ] E2E tests pass on a real OVH project: `./test/e2e/run-e2e.sh`
- [ ] Helm chart installs cleanly with `webhooks.enabled=true`
- [ ] CHANGELOG.md updated (auto-generated by `gh api` in CI)
- [ ] `metadata.yaml` includes the new release series if needed

## Cutting the release

```bash
# 1. Ensure clean working tree
git status   # should be clean

# 2. Tag the release (v-prefix required)
git tag -s -m "Release v0.1.0" v0.1.0
git push origin v0.1.0
git push github v0.1.0
```

Pushing the tag triggers the
[`release.yml`](../.github/workflows/release.yml) workflow, which:

1. Builds the multi-arch image (linux/amd64, linux/arm64) and pushes to
   `ghcr.io/rancher-sandbox/cluster-api-provider-ovhcloud:vX.Y.Z`
2. Packages the Helm chart and pushes to
   `oci://ghcr.io/rancher-sandbox/charts/cluster-api-provider-ovhcloud`
3. Runs `make release-manifests` to produce
   `infrastructure-components.yaml`, `metadata.yaml`, and the cluster
   templates
4. Creates a GitHub Release with auto-generated notes and uploads the
   manifests

## Post-release

- Announce the release in the project README and any communication
  channels.
- Bump `Chart.yaml` `version` and `appVersion` for the next dev cycle
  (if you want unreleased Helm pulls to work locally).
- Open a milestone for the next release.

## Local dry-run

To preview what the release pipeline would produce without pushing
anything:

```bash
make release RELEASE_TAG=v0.1.0-dryrun
ls -la out/
```

This runs `verify`, `test`, and `release-manifests`. It will fail if the
working tree is dirty.

## Hotfix releases

For urgent fixes:

1. Branch from the previous release tag: `git checkout -b release-0.1 v0.1.0`
2. Cherry-pick the fix onto the branch
3. Tag `v0.1.1` from the branch
4. Push the tag — same workflow runs

## Releasing the Helm chart only

The Helm chart is versioned independently of the controller. To release a
chart fix without rebuilding the image:

```bash
# Update chart/cluster-api-provider-ovhcloud/Chart.yaml version
helm package chart/cluster-api-provider-ovhcloud
helm push cluster-api-provider-ovhcloud-X.Y.Z.tgz oci://ghcr.io/rancher-sandbox/charts
```

(Or use the manual workflow trigger once we add it.)
